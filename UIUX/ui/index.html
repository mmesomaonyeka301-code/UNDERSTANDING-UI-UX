Session 01

User Interface Design — The Basics

basically, what even IS a UI?? 👀

A UI (User Interface) is literally just the way you interact with any tech — buttons, sliders, menus, text boxes. Think of your ATM screen or your phone — that whole visual layer? That's UI. It's what you SEE and TOUCH.

UI Design is the process of building those interactions so they're smooth, logical, and not chaotic. It mixes three things: Information Architecture (organizing stuff), Interaction Design (how things respond to clicks), and Visual Design (making it look slay).

The main parts of any UI are: Input Controls (buttons, checkboxes, dropdowns — how you give info), Navigation (breadcrumbs, search bars, icons — how you move around), Informational Components (tooltips, notifications, progress bars — how the app talks back to you), and Containers (like accordions that hide/show content).

There are 6 key design principles: Structure, Simplicity, Visibility, Feedback, Tolerance, and Reuse. The big ones to remember: Keep it simple (Apple vibes), make feedback clear (tell users what just happened!), and be tolerant of mistakes (like Dropbox's undo button — king behavior).

Types of interfaces include the old-school Command Line (MS-DOS era), Menu-Based (like ATMs), Natural Language (voice recognition, hey Siri!), Touch (your phone), Web-based, and the most popular one: GUI (Graphical User Interface) — windows, icons, cursors, the whole thing.

The UI design process is a spiral: Analyze → Design → Build → Validate, then repeat until it slaps. Models like Mental Model, Conceptual Model, and Interaction Model guide how designers think about what users expect.

Color Theory is also in here — primary, secondary, tertiary colors, warm vs cool tones, and color harmony (complementary, analogous, triadic). Colors literally affect user moods and behaviors, which is lowkey wild.

Big idea from Design Thinking: Empathize → Define → Ideate → Prototype → Test. You're not just making stuff look pretty — you're solving real human problems.

Session 02

User Experience Design — The Feel

UI is the knife. UX is the whole vibe of cutting the cake 🎂

UX (User Experience) is the WHOLE journey — not just what it looks like, but how it FEELS to use something. UI asks "what do I click?" UX asks "wait, why am I even here and does this make me happy?"

UI vs UX breakdown: UI is tangible, visible, and about HOW things look/work. UX is subjective, about WHY things are designed the way they are. UI is one part of UX. They're besties but different.

The 5 elements of UX Design: Visual Design (aesthetic vibes), Usability (can people actually use it?), Information Architecture (logical structure), Interaction Design (how things respond), and User Research (actually talking to real humans).

Usability has the "5 Es": Effective, Efficient, Engaging, Error Tolerant, Easy to Learn. A usable app does the job, doesn't waste your time, doesn't bore you, handles your mistakes gracefully, and doesn't need a PhD to figure out.

The UX Design Process goes: Research → Design → Test → Optimize. Research methods include interviews, surveys, analytics, competitor analysis, workshops, usability testing, and persona creation. Personas are basically fictional characters based on real data that represent your target users — they help designers stay focused on who they're building for.

Good UX best practices: don't make users think too hard, use familiar patterns, let people undo things, give clear feedback, keep menus simple, test continuously. Bad UX: cluttered layouts, confusing navigation, no feedback, heavy text walls. Nobody wants that energy.

Fun fact from the book: "Good design is actually a lot harder to notice than poor design" — Don Norman. When design is done right, it's literally invisible. You just feel good using it. Understood assignment.

Session 03

Responsive Web Design — It Bends, Not Breaks

your website should look fire on EVERY screen, no exceptions 📱💻

Responsive Web Design (RWD) is designing websites that automatically adjust to whatever screen size the user has — phone, tablet, desktop, whatever. The goal: one website, looks great everywhere.

RWD uses three core tools: Fluid Grids (layouts built with percentages, not fixed pixels), Flexible Images (images that scale with the container), and CSS3 Media Queries (code that says "if the screen is THIS wide, apply THESE styles").

Media queries are basically CSS superpowers. Example: @media only screen and (max-width: 768px) — this tells the browser "hey, if the screen is 768px or less, switch up the layout." CSS2 started it, CSS3 made it actually powerful.

Two ways to build it: write your own responsive CSS from scratch, or use a responsive stylesheet/framework. The book also covers Progressive Enhancement (start basic, add layers) vs Graceful Degradation (start fancy, fall back for older browsers). These are the two main philosophies for handling device/browser differences.

Benefits for users: better reading experience, less zooming and scrolling, faster load times. Benefits for designers and devs: one codebase, better SEO, lower maintenance costs. It's literally a W for everyone.

Session 04

RWD Strategies — How to Actually Do It

the real blueprint for building sites that don't flop on mobile 🔧

This session gets into the actual strategies behind building responsive sites. Key concept: Content Strategy — before you even think about layout, figure out WHAT content you need and WHY. A content audit is basically a full inventory of your site to see what's working, what's mid, and what needs to go.

Responsive vs Adaptive Design — these are different! Responsive is fluid and reacts to ANY screen size (CSS media queries doing the heavy lifting). Adaptive uses fixed layouts for specific screen sizes (320px for mobile, 760px for tablet, 960px for desktop). Responsive is more flexible; adaptive gives more control over exact layouts.

Mobile performance optimization is also covered: optimize images (don't load massive desktop images on mobile — huge L), use CSS background-image with media queries for swapping images, minimize unnecessary resources. Tools like Google PageSpeed Insights and WebPageTest help you measure and fix speed issues.

Web Accessibility is a BIG deal. About 15% of the world's population has some form of disability — that's a billion people. An accessible website means everyone can use it: people with visual, auditory, motor, cognitive, or intellectual disabilities. It's not just the right thing to do — it builds trust, expands your audience, and keeps you legally covered. Best practices include keyboard navigation, clear fonts, good color contrast, captions, and alt text for images.

Quick tips: plan for breakpoints at 320, 480, 760, 960, 1200, and 1600px. Use ems or percentages, not fixed widths. Test on actual devices, not just browser windows.

Session 05

RWD Patterns in Mobile Design

the actual design patterns that make mobile UX not suck 📐

This session is all about the practical patterns used in real mobile design. First up: Breakpoints. A breakpoint is the screen width where your layout switches up. Major breakpoints = big layout changes (like going from 2 columns to 1). Minor breakpoints = small tweaks (like moving a label). Every responsive site needs at least two breakpoints (tablet + mobile).

Navigation Drawer (aka hamburger menu) — that three-line icon that opens a side panel. Super popular on mobile because it hides nav items until you need them, saving screen space. Works great for complex sites with lots of pages.

Stacked Pagination — on mobile, content stacks vertically instead of spreading horizontally. This is the natural flow for small screens. Instead of side-by-side content, everything stacks on top of each other.

Fluid Images — images that stretch or shrink to fit their container. Use max-width: 100% to prevent images from overflowing their containers. Keeps things looking clean at any size.

Top Bar and Bottom Bar — fixed navigation elements at the top or bottom of the screen that stay visible as the user scrolls. Common in apps (think Instagram's bottom nav). Top bars usually have the logo and key actions; bottom bars have main navigation tabs.

Front Action Calls (FABs) — those big round floating action buttons (like the compose button in Gmail). They highlight the most important action on a screen. Bold, accessible, and impossible to miss.

Short and Simple Menus — keep menus minimal on mobile. Avoid deep nested menus that require too many taps. Users should find what they want in max 2-3 taps. Every extra tap is a potential dropout point.

Session 06

Usability Studies in RWD

find out if your design actually works before you ship it 🧪

Usability Studies = evaluating your product while real users actually use it. It's a type of "Black Box Testing" — you watch what users DO without caring about the internal code. You're judging the experience, not the engineering.

Two ways to run it: Laboratory Experiments (controlled environment, structured tasks) and On-Site Observation (watching users in their real environment). Both give you gold if you do them right.

The usability testing checklist covers three areas: Accessibility (can everyone use it?), Navigation (can people find things?), and Content (is the information clear and useful?).

When a problem is found, it gets rated by two factors: Impact (High/Medium/Low — does it block the task or just annoy?) and Frequency (how many users hit this issue?). Problems are then classified into Severity 1-4. Severity 1 = the app is broken, fix it NOW. Severity 4 = minor inconvenience, fix it eventually.

For RWD specifically: always test on ACTUAL mobile devices, not just a resized browser window. Roll-overs don't translate to touch screens. Overlays behave differently. Lighting conditions affect readability IRL. Don't skip real device testing!

Free tools for RWD testing: Responsive Test, Responsinator, Am I Responsive?, Viewport Resizer, Screenfly. Use them before you launch — your users will thank you.

Session 07

Understanding Figma

the design app that literally changed the industry 🎨

Figma is a browser-based UI/UX design tool — no downloads, no version conflicts, works on any OS. It's where modern designers live. Think of it as Google Docs but for designing websites and apps. It combines the best parts of Sketch, InVision, and Zeplin all in one place.

The Figma interface has 4 main areas: the Toolbar (top bar with all your tools — scale, move, shapes, text, pen, etc.), the Canvas (the big blank workspace where everything happens), the Layers Panel (left sidebar — shows all your elements, frames, groups, and assets), and the Properties Panel (right sidebar — Design tab, Prototype tab, and Inspect tab for viewing/editing element properties).

Key Figma concepts: Frames (like artboards — your screen/page containers), Components (reusable design elements — buttons, nav bars, cards — change one, change all), Variants (different states of the same component, like a button that's default vs hovered vs clicked).

Best features: real-time collaboration (multiple people can design together simultaneously — imagine Google Docs collab but for UI), intuitive prototyping (link frames together to simulate user flows without coding), sharing design feedback with comments directly on the design, plugin support for automating repetitive tasks, and Zeplin integration for handoffs to devs.

The Inspect Tab is literally a dev's best friend — it shows all the CSS values, measurements, colors, and typography specs of any element. Devs can look at a design and see exactly what CSS to write. Designer to developer handoff? Way smoother.

You can log into up to 10 accounts, access the Figma Community for free templates and plugins, and share prototypes via link with view or edit permissions. It's the whole package.

Session 08

Designing Websites with Figma

actually building stuff in Figma, hands on 🖥️

This is where everything comes together. Session 8 is about actually using Figma to design full websites — from wireframe to prototype. You're applying all the UI/UX principles from the earlier sessions inside the Figma tool.

Key workflow: Start with wireframes (rough layout sketches — don't care about colors yet, just structure and content placement). Move to mockups (add actual visual design — colors, fonts, imagery). Then build prototypes (link your frames together so you can simulate clicking through the site).

Component management in Figma is a whole skill. When you create a component (like a button), it becomes a "master component." Every time you use it across your design is an "instance." Change the master = all instances update. This is insanely powerful for maintaining consistent design systems.

The Instance Swap menu lets you swap one component for another (like swapping a Follow button for an Unfollow button) without rebuilding. Super efficient for designers handling large projects.

For web design specifically: set up your frame sizes for common breakpoints (desktop, tablet, mobile), use auto-layout for responsive components, leverage constraints to control how elements resize when frames change size, and use the Prototype tab to simulate navigation flows before any code is written.

The collaborative features really shine here — a designer and developer can literally be in the same Figma file at the same time. The dev can inspect elements, copy CSS values, and download assets without bugging the designer. The designer can make real-time updates that the dev sees instantly. No more "final_final_v3.sketch" files being emailed around. We're so back.
